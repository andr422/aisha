# Контекст проекта (для Claude Code) — обновление 4

## Что это
PWA для ежедневного сбора ABA-данных терапистами (русскоязычная ниша, аналоги
уровня TallyFlex). Три страницы:
- `final_fixed_aisha.html` — ежедневный сбор данных на занятии;
- `summary_fixed_aisha.html` — настройка протоколов/целей и профили детей;
- `abc.html` — ABC-запись поведения (эпизоды: время + A/B/C + длительность +
  интенсивность, ввод чипами).
Хостинг: GitHub Pages (репо andr422/aisha). Пользователь — не программист.

## Профили детей (реализованы; совместимость сверена — обновление 4)
Активен всегда ровно один ребёнок; его данные лежат под ПЛОСКИМИ рабочими
ключами (`protocol{i}_*`, счётчики, лог, `child_name`, `session_date`) — весь
код ввода их не различает. Переключение (только на сводной) сворачивает
рабочий набор текущего в `localStorage["aba_child_<id>"]` и разворачивает
выбранного. Список — `aba_children` [{id,label}], активный — `aba_active_child`.
История в IndexedDB — на ребёнка: ключ `aba_history::<id>`; первый профиль
наследует старую единую `aba_history`. При переключении с несданными
счётчиками — предупреждение, но переключение разрешено (данные остаются
в профиле неотправленными). Ежедневная показывает индикатор `#activeChild`
(«Ребёнок: <метка>») и берёт метку в отправку. ВЫВОД ПРОВЕРКИ: ежедневная
страница совместима со схемой профилей без префиксов — рабочие ключи плоские.

## Архитектура данных (ежедневная)
- История сессий: IndexedDB `aba_db` v2, стор `history`, ключ
  `aba_history::<activeChildId>`; кэш `historyCache`; фолбэк на localStorage
  (`window.__abaIdbFailed`); лимит 200 сессий; в таблице истории показываются
  последние 10.
- Очередь офлайн-отправки: стор `send_queue` (autoIncrement, FIFO); каждый
  запрос сначала в очередь, затем flushQueue(); авто-досыл по `online`,
  при старте, раз в 60 с; индикатор `#queueStatus`.
- Отправка: ТОЛЬКО итоги сессии по кнопке (SESSION_ENDPOINT). С июля 2026 —
  ЧЕСТНОЕ ПОДТВЕРЖДЕНИЕ ДОСТАВКИ (no-cors убран): sendPacket шлёт обычный
  CORS-запрос с Content-Type text/plain (JSON в теле; text/plain — чтобы не
  было preflight, который Apps Script не умеет; redirect: follow), читает
  ответ; доставлено = {ok:true} (или «OK» от старой версии скрипта). Отказ
  сервера → item.attempts++ (queuePut), пакет НЕ блокирует FIFO; после
  STUCK_ATTEMPTS=30 пропускается, индикатор «⚠ Не доставлено: N. Нажмите —
  повторить» (клик сбрасывает attempts). Сетевая ошибка → стоп до следующего
  flush. Отправка отдельных кликов УДАЛЕНА (июль 2026) — последовательность
  ответов уходит внутри итогов (`responses`). Скрипт-приёмник (Apps Script) сам
  раскладывает данные по таблицам детей: «ABA — <метка>» в папке «ABA — дети»
  на Диске владельца, по полю `child` пакета. Поэтому МЕТКИ ДЕТЕЙ НЕ
  ПЕРЕИМЕНОВЫВАТЬ без нужды (новая метка = новая таблица). Правки скрипта
  разворачивать ТОЛЬКО через «Управление развёртываниями → Новая версия»
  (не «Новое развёртывание» — иначе сменится URL).
- Счётчики текущей сессии: `aba_counts_v2` (по классам кнопок, объект
  {s,p,e} на блок) с авто-миграцией из старого позиционного `aba_counts`.
- `aba_response_log`, `aba_prompt_details` — в localStorage, по индексам блоков.
- При смене выбранных целей несданные счётчики/лог/подсказки мигрируют по
  именам целей (snapshotInProgressData/restoreInProgressData), undo-стек
  очищается; история мигрирует по именам (migrateSessionDataForProtocol).

## ABC-запись (abc.html)
- Эпизод: время (input time, авто «сейчас») + чипы B (обязательно), A, C
  (одиночный выбор; «✎ вписать» — ПЕРВЫЙ чип группы, свой текст через
  prompt), длительность (5 пресетов), интенсивность (низкая/средняя/высокая).
- Списки A/B/C настраиваются на самой странице («⚙ Списки», textarea по
  строке на вариант), ключи abc_antecedents/abc_behaviors/abc_consequences;
  пустой список = дефолты. Журнал: abc_log (localStorage, лимит 500,
  показываются последние 30, удаление ✕ — только локально).
- ВСЕ ключи abc_* входят в рабочий набор ребёнка (isChildDataKey на сводной:
  /^abc_/) — списки и журнал у каждого ребёнка свои.
- Отправка: сразу при записи эпизода, через ту же очередь send_queue на
  SESSION_ENDPOINT с payload {type:"abc", child, date, time, behavior,
  antecedent, consequence, duration, intensity}. Скрипт-приёмник различает
  пакеты по type и пишет эпизоды на лист «ABC» таблицы ребёнка.
- Экспорт журнала в CSV — кнопка на странице.

## Метрика (НЕ менять без явного запроса)
Процент = самостоятельные + / (+ P −). Подсказки и ошибки одинаково
в знаменателе — осознанное решение (% независимых реакций).

## Ввод и UI (актуальная версия ежедневной)
- КНОПКИ в ABA-нотации: «−» слева (ошибка), «P» центр (подсказка),
  «+» справа (самостоятельно). Знаки типографские (− это U+2212),
  display-шрифтом, в цвет чипа. Порядок — по числовой оси.
- Вся логика счёта — ПО КЛАССАМ кнопок (btn-success/btn-prompt/btn-error,
  хелпер countOf), НЕ по позициям.
- Имя цели — в `block.dataset.goalName` (в .goal-label может быть бейдж
  «освоено»; textContent метки для отправки НЕ использовать).
- Короткий тап P = обобщённая подсказка (код P); долгое нажатие 450мс =
  модалка типов G/V/E/PF/FF (словами). Undo-стек. Режим «Заблокировано».
- Вибрация 20мс + CSS-вспышка .flash на каждой записи (increment).
- ЖЕСТОВОГО ВВОДА НЕТ (был реализован и осознанно удалён: перестановка
  кнопок сделала его избыточным). Не восстанавливать без запроса.
- Режимы вида `aba_view_mode`: "list" | "single". Single: те же DOM-блоки,
  скрыты CSS; кнопки −/P/+ ГИГАНТСКИЕ; навигация «Предыдущая/Следующая»;
  бейдж «цель i из n».
- Спарклайн прогресса: последние 10 сессий, buildSparkline(), цвета по
  порогам ≥80 / ≥50; бейдж «освоено» = 3 последние сессии ≥80 (checkMastery).
- Пустые протоколы (ключ с пустым названием) не рендерятся; сводная удаляет
  ключ при стирании значения.
- Сводная рендерит протоколы/цели ДИНАМИЧЕСКИ: только заполненные, плюс
  кнопки «＋ Добавить протокол» и «＋ Добавить цель» — БЕЗ ограничений
  количества (номера берутся max+1 по ключам). Пустая добавленная карточка
  исчезает при перерисовке (самоочистка). Схема ключей protocol{i}_* не
  менялась.
- «🗑 Удалить протокол» (deleteProtocol): удаляет ключи протокола k,
  ПЕРЕНУМЕРОВЫВАЕТ протоколы выше (k+1→k…, moveProtocolKeys) и вырезает
  колонку k из истории активного ребёнка (spliceHistoryProtocol: IndexedDB
  aba_history::<id> + localStorage-фолбэк) — история позиционная, дыры
  недопустимы. Несданные счётчики сессии при удалении СБРАСЫВАЮТСЯ
  (о чём предупреждает confirm). Отправленное в Google не трогается.
- PDF протокола: кнопки «📎 PDF / 📄 Открыть / заменить / ✕» в шапке карточки
  на сводной; на ежедневной у протокола с файлом появляется «📄 протокол»
  (класс .pdf-open, не .btn — виден и в режиме «по одной»). Хранение:
  IndexedDB, стор history, ключ aba_pdf::<childId>::p<i>, лимит 20 МБ на файл;
  в Google НЕ отправляется, в экспорт настроек НЕ входит. При перенумерации
  протоколов PDF переезжает (pdfMove), при удалении профиля — чистится
  (deleteChildPdfs). Просмотр: objectURL + window.open.
- ШАБЛОНЫ протоколов: ключ `aba_templates` (ОБЩИЙ для всех детей, НЕ в
  рабочем наборе) — [{title, goals}]. Наполнение: «💾 В шаблоны» на карточке
  протокола ИЛИ раздел «📚 Шаблоны» (кнопка в тулбаре сводной; редактор:
  первая строка — название, дальше цели по строкам; список с ✎/✕).
  Автоподстановка: при совпадении названия (normTitle: trim+lowercase) у
  протокола БЕЗ целей появляется «⚡ Подставить цели из шаблона (N)»
  (refreshTemplateHint) — только по кнопке, ничего не перетирает. Шаблоны
  входят в экспорт (payload.templates), при импорте домешиваются (upsert).
- Экспорт/импорт настроек (формат _version:2) МОЖЕТ включать PDF: при экспорте
  спрашивает (файл тяжелее), внутри JSON поле pdfs={номерПротокола:dataURL}.
  Импорт восстанавливает их в IndexedDB активного ребёнка (restoreImportedPdfs,
  сперва чистит старые). Файлы v1 без pdfs импортируются как раньше.
- Вырезанная история НЕ пропадает: уходит в локальный АРХИВ ребёнка —
  ключ `aba_archive` (в рабочем наборе, лимит 50 записей), запись
  {archivedAt, title, goals, sessions:[{date, values, prompts}]}. Раздел
  «Архив удалённых протоколов» внизу сводной (renderArchive, вызывается из
  renderSummary): по записи — «⬇ CSV» (Дата;Цель;Процент;Подсказки) и
  «✕ стереть» (окончательно, с confirm).
- Ежедневная: existingGoalIds() — цели протокола из ключей (без потолка 10);
  выбор целей показывает только существующие; getSelectedGoalIds без явного
  выбора возвращает ВСЕ существующие цели (не [1,2,3]) и отфильтровывает
  удалённые.

## Дизайн (текущий, «B2-elevated»)
- Палитра в :root: море (--ok*, успех) / песок (--prompt*, подсказка) /
  камень (--err*, ошибка — нейтральный серый, НЕ красный); низкий процент
  данных — кирпичный --low-ink; фон тёплый #f3f2ed; --accent тёмная бирюза.
- Процент — герой карточки: .block-head (имя слева, .block-stats справа:
  .percent 34px/900 + спарклайн). Кнопки-«фишки»: нижняя тень 3px,
  :active утапливание, анимация .flash при записи.
- Шрифты: Unbounded (display) + Onest (body) через @font-face с фолбэком на
  системные; файлы ожидаются в fonts/Unbounded.woff2 и fonts/Onest.woff2
  (пользователь мог ещё не положить — это нормально).

## Общие модули (подключаются на каждой странице перед </body>)
- tabbar.js — нижний таб-бар: Сбор · Таймер · Настройка. Список вкладок —
  массив TABS в начале файла; новый чек-лист = один объект в массив.
  Поддерживает вкладки-действия (поле action вместо href).
- timer.js — визуальный таймер (тающее кольцо): пресеты 3с–5м, пауза по
  тапу, звук (WebAudio, ключ aba_timer_sound), вибрация, мини-таблетка.
  Состояние в localStorage (aba_timer_state) по метке времени — переживает
  переходы между страницами. Подключать ПЕРЕД tabbar.js.
- Порядок подключения: <script src="timer.js"></script> затем
  <script src="tabbar.js"></script>. Старой плавающей кнопки навигации
  (.nav-left-floating) больше нет ни на одной странице.

## Известные ограничения / планы
- Полный бэкап планшета одним файлом (история всех детей + ABC-журналы +
  архивы + PDF) — следующий кандидат; текущий экспорт настроек историю
  замеров НЕ включает.
- Старое развёртывание скрипта итогов (`AKfycbx1ZyMrh…`, пишет в архивную
  таблицу Aisha) держится до обновления всех планшетов, потом архивировать.
- Меню «⋮» справа в шапке ежедневной (menu-wrap/wireMenu): «Вид: список/по
  одной» (#viewModeBtn живёт в меню), «📊 Графики», «⬇ Экспорт (CSV)»,
  «⬇ Отчёт с графиками». «Отменить действие» и «Отправить данные» — вне меню
  (нужны мгновенно). Закрытие: клик мимо, Esc, после любого пункта.
- Экспорт истории замеров: пункты меню «⋮».
  «⬇ Экспорт (CSV)» — вся история активного ребёнка (до 200 сессий) в длинном
  формате «Дата;Протокол;Цель;Процент;Подсказки», UTF-8 с BOM, разделитель
  «;» (для русского Excel), пустые замеры пропускаются. «⬇ Отчёт с графиками» —
  самодостаточный HTML-файл для родителей/супервизора (SVG-графики по целям,
  buildReportHtml). Кнопка «📊 Графики» — раздел графиков прямо в приложении
  (renderCharts). Общая основа: collectChartData + buildChartSvg (шкала
  0–100, пунктирные пороги 80/50, цвета точек как у спарклайна).
- Нет графиков крупнее спарклайна, нет других методов записи (частота,
  длительность) — направление развития по образцу TallyFlex.

## Стиль работы
Пользователь общается по-русски; комментарии в коде по-русски. Перед крупными
переделками — обсуждать варианты (пользователь любит выбирать по макетам).
Точечные правки предпочтительнее рефакторинга. После правок ежедневной —
всегда напоминать про версию sw.js и предупреждение тераписту, если меняется
моторика ввода.
